Splunk gives the Application Support Analyst (ASA) Team real-time access to the WebLogic application error logs to troubleshoot issues and work with the Dev Team to provide detailed context to issues. It also provides the ability to create reporting and error tracking by Splunk alerting based on application errors. Splunk is utilized by the ASA Team to create pre- and post-deployment error baselines that allow a quick response to developing programmatic errors. It also allows for error trend data, both application-wide and error-specific. Splunk alerting can be set up by the ASA Team to allow automated tracking of trending application errors. Auditing Build audits are performed to verify the product matches. The Configuration Items (Cis) described in the specification, other documents, and the package is reviewed before release. The audits and reviews vary in complexity and formality. At a minimum, each named build goes through the following audits: A comparison of the Version Description Document (VDD) and the build change log. A comparison of the VDD and query of Jira or GitHub work items tagged with the build label. A review of the change log and work items contained in the build to ensure that code commits were targeted to the appropriate repository stream. A review of Jira or GitHub work items to ensure appropriate detail and work item state and traceability to requirements and test cases are captured. Configuration audits are broken into functional and physical configuration audits. They occur either at delivery or now of effecting the change. Configuration audits are functional and physical. A functional configuration audit ensures functional and performance attributes of a CI are achieved. A physical configuration audit ensures a CI is installed in accordance with the requirements of its detailed design documentation. Security Strategy Transport Layer Security (TLS) is present leveraging Secure Sockets Layer (SSL) certificates issued from the VA certificate authority. For the connection to services, such as CEST API application, the client (Awards) and the service have a shared secret that is used to verify the Java Web Token (JWT). Both client and services store their secrets in Vault.
